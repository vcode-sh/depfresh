# depfresh 2.0.1 Release Design

## Goal

Publish the completed first-class workspace and catalog exclusion work as `depfresh@2.0.1`
through the repository's existing immutable-tag release workflow, then prove that npm and GitHub
expose the exact hosted artifact.

## Current truth

- Local `main` is a normal checkout at `f3649cb`, one reviewed Plan 028 commit ahead of
  `origin/main`; the worktree is clean.
- npm `latest` is `2.0.0`; npm has no `2.0.1` version.
- GitHub has no `v2.0.1` tag or release.
- `.github/workflows/release.yml` accepts only `v${package.json.version}`, rebuilds under exact
  Node `24.15.0`, installs isolated npm `11.12.0`, runs the complete gate, publishes with trusted
  OIDC, verifies the public package, and creates a GitHub release from a repository release note.

## Release candidate

The release candidate will:

- bump `package.json` and every maintained exact-version runner example from `2.0.0` to `2.0.1`;
- move the Plan 028 changelog entry from `Unreleased` into a dated `2.0.1` section;
- add `docs/releases/v2.0.1.md` without changing `docs/releases/v2.0.0.md`;
- point the hosted release job at the new release note;
- keep package formats, Node and package-manager floors, schema versions, APIs, and runtime behavior
  unchanged from the reviewed Plan 028 commit.

## Verification and publication

Before any push or tag, the candidate must pass the exact release workflow commands locally with
disposable HOME/cache/store/artifact state. The actual tarball must install and pass the packaged
verifier under Node `24.15.0` and npm `11.12.x`.

After the candidate commit is pushed, hosted `main` CI must complete successfully before an
annotated `v2.0.1` tag is created and pushed. The tag workflow is the only publisher. Completion
requires a successful hosted release run, npm `latest=2.0.1`, matching local/hosted/public
integrity, an isolated public install, a non-draft non-prerelease GitHub release with the exact
tarball, and provenance bound to the tag workflow and release commit.

## Failure boundaries

- Never publish locally or bypass the trusted workflow.
- Never create or move a floating `v2` tag.
- Never force-move `v2.0.1`; if an immutable tagged candidate cannot publish safely, stop and
  diagnose rather than rewriting public tag identity.
- Do not claim production/public completion from local or hosted verification alone.
