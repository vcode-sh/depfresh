# depfresh 2.1.2 Hardening Design

Status: approved for autonomous implementation by the user on 2026-07-20.

## Goal

Release depfresh 2.1.2 with the audited release, cache, security, contract, documentation, test,
CI, dependency-policy, and validation-performance findings resolved without weakening any existing
fail-closed behavior or changing the Node 24.15.0 runtime baseline.

## Scope

The release includes every concrete P0-P3 finding from the post-Plan-038 audit:

1. Consume npm 11 array and npm 12 keyed-object pack manifests through one tested helper in every
   release workflow phase.
2. Make registry cache identities free of credentials and authorization-safe.
3. Keep operational SQLite failures on the documented memory fallback path and preserve the
   authoritative command result when cache statistics or close fail.
4. Redact arbitrary human and debug error messages at their final output boundary.
5. Accept only an exact GitHub `owner/repository` identity before constructing authenticated API
   requests.
6. Bound successful registry response bodies and the aggregate GitHub tag traversal.
7. Freeze the exact bytes of every published v1 schema.
8. Correct maintained Plan 038, npm 12, and release-version truth surfaces while leaving historical
   evidence unchanged.
9. Isolate the small process-observation test surface while making ordinary tests parallel and
   retry-free.
10. Reuse the tested distribution in CI, add an installed-package Windows read-only lane, and stop
    Dependabot from proposing an incompatible `@types/node` major.
11. Pre-index demonstrated hot semantic-validation lookups and add a scale regression without
    altering validation order or ambiguity handling.
12. Publish and independently verify version 2.1.2 from an exact tag and exact tarball.

Yarn manager execution and private-registry artifact verification remain separate design projects.
They are not part of this release.

## Architecture

### Release manifest boundary

`scripts/pack-manifest.mjs` remains the only parser for `npm pack --json`. A small executable helper
will expose selected validated fields to shell workflows, avoiding duplicated inline JSON parsing.
The helper must reject empty, multiple, mismatched, or missing-field manifests before any output is
written.

### Cache boundary

Registry URLs are normalized to an origin plus pathname identity with userinfo, query, and fragment
rejected or removed before persistence. Authenticated registries bypass persistent cache reads and
writes; secrets are never hashed into or persisted as cache keys. SQLite owns a memory-cache
fallback that activates after an operational database failure. Cache diagnostics remain best-effort,
and closing resources cannot replace the command's authoritative result.

### Input and output security

All arbitrary errors cross `getSafeErrorDetails()` or `redactSensitiveText()` immediately before
human/debug output. GitHub specs must contain exactly two safe path segments. Successful HTTP bodies
are read through a byte-limited reader before JSON parsing. GitHub pagination additionally has fixed
page, record, and elapsed-time budgets; exceeding any budget is an error/unknown result, never a
partial success.

### Compatibility and performance

Every shipped v1 schema receives an exact SHA-256 guard sourced from the last published bytes. The
validator constructs uniqueness-aware maps and grouped boundary indexes once, but retains canonical
array ordering and every duplicate/ambiguity rejection. A deterministic scale test proves bounded
growth by operation counts or a generous fixed-clock envelope without becoming machine-flaky.

### Test and CI topology

The default Vitest project is parallel and has `retry: 0`. Only explicitly enumerated tests that
observe same-user processes run in a serial project; any retry is limited to that project and exact
documented reason. CI builds once, uploads the tested `dist`, and downstream distribution jobs use
those exact bytes. Windows installs the packed tarball and tests read-only entry points only; manager
execution stays unsupported.

## Error handling

- Cache fallback records a sanitized debug diagnostic and never turns unknown into success.
- Registry limits use stable `RegistryError` or `ResolveError` reasons and cancel unread bodies.
- Invalid GitHub identities are treated as unsupported dependency specs and never trigger a request.
- Release-manifest ambiguity fails before outputs, publication, or hosted release creation.
- Public release verification compares the registry integrity with the exact verified tarball.

## Verification

Each behavior follows RED/GREEN TDD with focused Vitest coverage. The final release gate uses Node
24.15.0, pnpm 11.15.1, npm 12.0.1 where the release workflow requires it, schema checks, TypeScript,
Biome with zero warnings, the complete retry-free suite with coverage, release tests, practical and
WUN-shaped demos, build, dry-run pack, exact packed-package verification, and installed replay.
After push, hosted CI must pass before tagging. The tag workflow must publish the exact tarball,
create the curated GitHub release, and expose matching public npm integrity.

## Compatibility

No public JSON schema v1 bytes, library exports, CLI exit codes, manager authority requirements, or
artifact-trust semantics may change. Node remains locked to 24.15.0. All other direct dependencies
remain on their current compatible latest versions; `@types/node` remains on major 24 by policy.
