# depfresh 2.1.2

Release date: 2026-07-21

Status: Source text alone does not establish publication; npm and GitHub external records are
authoritative.

This release carries the completed [Plan 038](../../plans/038-visual-plus-hybrid-default.md) Visual+
experience into a hardened release pipeline. It also completes the approved
[2.1.2 hardening plan](../superpowers/plans/2026-07-20-2.1.2-hardening.md) without changing the
published machine schemas or authority model.

## Visual+ human output

- Plan 038 makes the five-region hybrid review the locally proven default for eligible human table
  runs: context, overview, risk focus, a complete update ledger, and the final receipt.
- Every selected update appears exactly once in the ledger. `--long` remains the exhaustive view of
  operations, owners, shared dependencies, occurrences, physical targets, lifecycle, and receipts.
- Capable, narrow, plain, piped, CI, and `TERM=dumb` routes preserve the same evidence and failure
  truth. Interactive, JSON, global, silent, veto-capable, and library routes keep their separate
  established contracts.

## Release and CI integrity

- npm 11 and npm 12 pack manifests now pass through one strict, version-independent reader before
  filename or integrity values enter release automation.
- CI builds `dist` once and reuses the exact tested bytes in downstream build, Visual+ replay, and
  distribution-smoke jobs. The verified tarball, not a rebuilt checkout, feeds the Windows proof.
- Process-observation tests are isolated in one explicit serial Vitest project. The default project
  remains parallel with zero retries, and aggregate coverage still spans both projects.
- Windows verifies the installed command shim with version, help, capabilities, and an unchanged
  read-only fixture; manager execution is neither authorized nor requested.
- Current official Actions are pinned by full commit SHA. Dependabot ignores only major updates for
  `@types/node`, so Node 24 type fixes continue to arrive.

## Security and resilience

- Credential-safe registry caching keeps authenticated npm and GitHub requests out of persistent
  storage and cross-authorization in-flight reuse. Anonymous cache identities exclude URL userinfo,
  queries, fragments, and credential material.
- SQLite cache operations permanently fail over to the owned memory cache after an operational
  error. Cleanup and statistics failures cannot replace a successful command result or its original
  error.
- Central error redaction protects CLI, debug, discovery, and orchestration output. GitHub package
  references must contain exactly one conservative owner segment and one repository segment before
  token access or a request.
- Bounded registry response processing caps successful npm, JSR, and GitHub JSON bodies. GitHub
  traversal also has page, record, and monotonic aggregate-time limits; a breached boundary returns
  no partial candidates.

## Compatibility and performance

- Published v1 schema bytes are guarded by exact hashes for inspect, plan, apply, error,
  global-plan, global-apply, and capabilities contracts.
- Maintained documentation now matches npm `>=10.0.0 <13.0.0`, pnpm `>=10.0.0 <12.0.0`, Bun
  `>=1.2.0 <2.0.0`, and public npm artifact verification with npm
  `>=11.12.0 <12.0.0 || >=12.0.0 <12.1.0`.
- Indexed semantic plan validation builds immutable occurrence, decision, boundary, owner/name, and
  cohort lookups once. It preserves canonical array order, duplicate rejection, ambiguity handling,
  first/last-match semantics, and every fail-closed result while removing repeated full-plan scans.

## Proof boundary

The tag workflow creates the hosted release only after the aggregate suite, exact npm 12 artifact
verification, and exact public-integrity gates succeed. Local evidence describes source readiness
only; this document neither establishes nor preclaims npm or GitHub availability.
